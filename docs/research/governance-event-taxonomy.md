## Envelope (shared)

Every event written by `spine.append` carries the standard envelope:

```jsonc
{
  "v": 1,
  "id": "evt_<14-digit-ts>_<6-hex>",   // generated by spine.genId
  "ts": "<ISO-8601 UTC>",
  "type": "<TYPE>",                     // one of EVENT_TYPES
  "actor": "ses_…" | null,              // appending session, if any
  "lane": "<lane-id>" | null,
  "data": { /* type-specific, documented below */ },
  "triggered_by": { "kind": "…", "id": "…", "fired_at": "<ISO>" } // optional
}
```

`triggered_by` provenance is required on any event a schedule, gate, or
auto-approval emits on the agent's behalf — see Phase 4 trigger discipline.

## Payloads

### 1. `GOAL_DECLARED`

```jsonc
data: {
  "objective":   "string — the agent's goal in one sentence",
  "constraints": ["string", "string", "..."]   // 0..N, optional
}
```

Latest event wins for the `goal` projection slot. No referential constraint.

### 2. `PHASE_DECLARED`

```jsonc
data: {
  "name":  "string",                  // free-form, e.g. "Phase 2 — gates"
  "notes": "string | null"            // optional context
}
```

Latest event wins for the `phase` projection slot.

### 3. `GATE_RAN`

```jsonc
data: {
  "gateId":     "kebab-case-id",                          // e.g. "tracked-source-drift"
  "ok":         true | false,
  "severity":   "critical" | "safety" | "warn",
  "durationMs": 1234,
  "evidence":   { /* gate-specific JSON, may be null */ } | null
}
```

`verify-spine` checks `gateId` resolves to a discoverable gate at verify
time as a **warning** only. Operator gates may be removed between runs
without invalidating prior `GATE_RAN` history.

### 4. `SOURCE_HASH_RECOMPUTED`

```jsonc
data: {
  "count": 3,
  "paths": [
    { "path": "docs/hard-rules.md",  "hash": "sha256-hex" },
    { "path": "CLAUDE.md",           "hash": "sha256-hex" },
    { "path": "docs/02-concepts.md", "hash": "sha256-hex" }
  ]
}
```

Emitted by `maddu sources rebuild`. Replayed into the `sourceHashes`
projection slot — latest `SOURCE_HASH_RECOMPUTED` per path wins.

### 5. `SLICE_SCOPE_DECLARED`

```jsonc
data: {
  "sliceId":         "string — operator-chosen or auto-derived from session",
  "scope":           ["path1", "path2", "..."],
  "lockedScopeHash": "sha256-hex over JSON.stringify(scope.sort())",
  "expansionBound":  { "maxFiles": 5, "maxGrowthPct": 30 }
}
```

### 6. `SLICE_SCOPE_EXPANDED`

```jsonc
data: {
  "sliceId":    "string — must match prior SLICE_SCOPE_DECLARED.sliceId",
  "addedPaths": ["pathN", "..."],
  "newHash":    "sha256-hex over JSON.stringify((scope ++ addedPaths).sort())",
  "reason":     "string"
}
```

`verify-spine` rejects an orphan `SLICE_SCOPE_EXPANDED` (no matching
`SLICE_SCOPE_DECLARED.sliceId` in prior events) as a referential-integrity
failure.

### 7. `SLICE_FUNCTIONAL_APPROVED`

```jsonc
data: {
  "sliceId": "string — must match prior SLICE_SCOPE_DECLARED.sliceId"
}
```

After this event for a given `sliceId`, the `slice-scope` gate refuses
non-doc / non-state edits on that slice until a fresh `SLICE_SCOPE_DECLARED`.

### 8. `TRIGGER_FIRED`

```jsonc
data: {
  "triggerId":  "string",                    // matches an entry in triggers.json
  "target":     "string — top-level command name",
  "cooldownMs": 60000
}
```

Always carries `triggered_by`. Replayed into the `triggers` projection
slot as `{ lastFiredAt, cooldownMs }` keyed by `triggerId`.

### 9. `PENDING_ACTION_ENQUEUED`

```jsonc
data: {
  "actionId": "act_<14-digit-ts>_<6-hex>",   // mirrors event id shape
  "kind":     "review" | "drift-check" | "brief-refresh" | "string",
  "payload":  { /* kind-specific JSON */ }
}
```

### 10. `PENDING_ACTION_DRAINED`

```jsonc
data: {
  "actionId": "act_… — must match prior PENDING_ACTION_ENQUEUED.actionId",
  "outcome":  "ok" | "skipped" | "error",
  "detail":   "string | null"
}
```

`verify-spine` rejects orphan DRAINED events as a referential failure.

### 11. `SLICE_REVIEWED`

```jsonc
data: {
  "sliceEventId":    "evt_… — must match prior SLICE_STOP.id",
  "verdict":         "CLEAN" | "P1" | "P2" | "P3" | "INFO",
  "findingsCount":   3,
  "reviewerRuntime": "runtime-name",
  "reviewPath":      ".maddu/reviews/<slice-event-id>.md"
}
```

### 12. `FOLLOWUP_OPENED`

```jsonc
data: {
  "fromReviewEventId": "evt_… — must match prior SLICE_REVIEWED.id",
  "severity":          "P1" | "P2" | "P3",
  "draftScope":        ["path1", "path2", "..."]
}
```

## v0.17 agent-native bootstrap

Four event types are reserved in Phase 0 and emitted across Phases 1–5
of the agent-native bootstrap rollout. They participate in the same
forward-compat discipline as the governance-layer types above — Phase 0
reserves the names in `EVENT_TYPES` so older spines remain readable
when projections that consume them are introduced later.

`SESSION_REGISTERED.data` is also extended (optionally — old events
without it remain valid) with a `parentSessionId` field used by
Phase 2's `sessionsTree` projection slot. Verify-spine treats absence
as `null`; if present, it must reference a prior `SESSION_REGISTERED`
event id in the same spine.

### 13. `SESSION_AUTO_REGISTERED`

```jsonc
data: {
  "sessionId":        "ses_… — id of the newly-registered session",
  "parentSessionId":  "ses_… — optional; must reference prior SESSION_REGISTERED or SESSION_AUTO_REGISTERED",
  "source":           "cli" | "spawn" | "agent-bootstrap",
  "label":            "<one-line, auto-derived from cwd-basename when source=cli>",
  "role":             "implementer" | "<...>"
}
```

Emitters:

- `maddu register` → `source: 'cli'`.
- `spawnWorker` when the runtime descriptor has `autoRegister: true` →
  `source: 'spawn'`, `parentSessionId` set to the caller's session id.
- Agent-first-read bootstrap (future) → `source: 'agent-bootstrap'`.

Idempotency: when `MADDU_SESSION_ID` is set in env AND that session is
still open in the projection, `maddu register` emits nothing (no-op).

### 14. `SESSION_STALE_DETECTED`

```jsonc
data: {
  "sessionId":        "ses_… — must match a prior SESSION_REGISTERED or SESSION_AUTO_REGISTERED",
  "lastHeartbeatAt":  "ISO-8601 — last SESSION_HEARTBEAT.ts for this session, or registration ts if none",
  "ageMs":            123456
}
```

Emitted by the inline janitor (Phase 5) when an open session's last
heartbeat exceeds the stale threshold (default 30 min, configurable
via `.maddu/config/janitor.json`). One event per detection; the
projection's `janitor.staleSessions` slot deduplicates.

### 15. `SESSION_AUTO_CLOSED`

```jsonc
data: {
  "sessionId":        "ses_… — must match a prior SESSION_REGISTERED or SESSION_AUTO_REGISTERED",
  "reason":           "janitor-stale",
  "lastHeartbeatAt":  "ISO-8601"
}
```

Emitted by the janitor when an open session ages past the auto-close
threshold (default 4 hours). MUST carry a `triggered_by` envelope per
the trigger-discipline gate; the janitor entry is shipped in the
default `.maddu/config/triggers.json` allowlist.

### 16. `AGENT_FILE_SYNCED`

```jsonc
data: {
  "files":  ["MADDU.md", "CLAUDE.md", "AGENTS.md"],
  "action": "create" | "merge" | "no-change"
}
```

Emitted by `maddu init` and `maddu upgrade` after the marker-delimited
merge into the three repo-root agent files completes. One event per
init/upgrade run (not per file). `action: 'no-change'` is emitted when
all three files were already in sync — preserved so re-runs leave a
trace without churning the spine with new content.

## Non-goals (per plan §3.8)

This taxonomy contains **no** DPS-domain content: no "5 Laws", no
Surface/Layer/Behavior governance vocabulary, no IntentExecutor, no
hard-coded numbered invariants. Everything here is operator-extensible
content placeholders the framework merely names and validates.

## Reservation rationale

These types are added in Phase 0 — **before** any emitter ships — so
that:

1. `verifySpine` accepts the names from day one. A repo that never
   emits any of them remains identical to a baseline repo.
2. Older installs (pre-governance) running a newer CLI never see
   "unknown event type" errors when reading newer spines after upgrade.
3. The rollout order (1 → 7) can land slices with no breaking-compat
   step between them.

## Mutability

This taxonomy is append-only after Phase 0. Adding a new type later
goes through the normal "new type + new arm + verify-spine + projection
slot" loop. Renaming or removing a reserved type requires a deprecation
window and a migration note in `CHANGELOG.md`.
