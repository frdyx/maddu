## Items

### #15 — Cross-stack tool detector (P3)

`maddu test/format/lint` against a Python project (`pyproject.toml` +
pytest + ruff) all return `refused  no-detector`. v1.1.0 marketing
implies cross-stack but the resolver is npm-only.

**Suggested fix:** extend `detectFramework()` in
`template/maddu/runtime/lib/tools.mjs` to recognize:

- `pyproject.toml [tool.pytest.ini_options]` → `pytest`
- `pyproject.toml [tool.ruff]` → `ruff check`
- `pyproject.toml [tool.black]` → `black .`
- `pyproject.toml [project.scripts]` → script-based fallback

Acceptance: a fresh Python repo can run `maddu test` and see pytest run
under the audited subprocess wrapper, with `TOOL_INVOKED`/`TOOL_COMPLETED`
events on the spine.

### #20 — Skill-candidate pattern threshold (P3)

After 1 slice-stop + 311 events + multiple commits the candidate detector
emitted 0 candidates. Pattern threshold (N=3) is too tight for short
runs.

**Suggested fix:** lower the threshold to N=2 for runs with fewer than 5
slice-stops, OR add a `--surface-all` flag to skill candidates listing
so the operator can inspect sub-threshold patterns manually.

### #12 — Compress consumer-install `upgrade` refusal text (P3)

`./maddu/run upgrade` from inside a consumer install refuses with "is NOT
a framework source" — correct, but the multi-line error is wordy. Could
compress to a one-line pointer to the `npx github:frdyx/maddu upgrade`
form.

**Suggested fix:** in `commands/upgrade.mjs` (or wherever the refusal
lives), trim the message to one line + the npx hint.
